## 🔍 QA Review — PR #313: Santander Provider

### Resumo

PR adiciona integração completa com o Santander ao módulo multi-provider de pagamentos do ERP. Inclui: autenticação mTLS + OAuth 2.0 com cache de token, registro de boleto híbrido (barras + QR Code Pix), consulta de status com dados de liquidação, cancelamento (baixa), webhooks para notificação de pagamento, download de PDF, sequenciamento atômico do nosso número, e frontend de configuração com upload de certificado PEM. Integra-se ao registry/factory existente sem quebrar providers anteriores.

### Findings

---

#### Finding 1
- **Severity:** 🟡 WARN
- **Confidence:** 0.85
- **Arquivo:** `santander.provider.ts` — função `paidAmount` no `parseWebhookEvent` (~linha 390)
- **Descrição:** A heurística para converter valor pago de reais para centavos é frágil. O código usa `numValue < 1000 && paymentValue.toString().includes(".")` para decidir se converte. Um boleto de R$ 999,00 (sem centavos, representado como `999`) não seria convertido para centavos. Pior: um valor como `500.0` (que é 500 reais) seria convertido para 50000, mas `1500.50` (1500 reais e 50 centavos) NÃO seria convertido (> 1000), resultando em `paidAmount: 1501` (centavos) quando deveria ser `150050`.
- **Correção sugerida:** Definir uma convenção fixa. Se o Santander sempre envia em reais com decimais, sempre multiplicar por 100. Se há ambiguidade, logar warning e documentar. Sugestão:
```typescript
// Santander API retorna valores em reais (ex: "150.50")
// Sempre converter para centavos
paidAmount = Math.round(numValue * 100);
```

#### Finding 2
- **Severity:** 🟡 WARN
- **Confidence:** 0.80
- **Arquivo:** `santander.provider.ts` — `parseWebhookEvent` (~linha 350)
- **Descrição:** O campo `bankNumber` no webhook é extraído com fallback para `nsuCode`: `payload.bankNumber ?? payload.nossoNumero ?? payload.nsuCode`. O `nsuCode` NÃO é o `bankNumber` — são identificadores diferentes usados em partes distintas do gatewayId. Se o webhook mandar apenas `nsuCode` sem `bankNumber`, o `buildGatewayIdFromWebhook` vai montar um gatewayId com o valor errado, fazendo a busca do boleto falhar silenciosamente.
- **Correção sugerida:** Remover `payload.nsuCode` como fallback para `bankNumber`:
```typescript
const bankNumber = (
  payload.bankNumber ??
  payload.nossoNumero ??
  ""
).toString();
```

#### Finding 3
- **Severity:** 🟡 WARN
- **Confidence:** 0.75
- **Arquivo:** `santander-auth.ts` — `authenticatedFetch` (~linha 130)
- **Descrição:** Quando recebe 401, o método invalida o token e lança exceção. Porém, não faz retry automático. O caller recebe o erro e o boleto falha. Em cenários de race condition (token expirou entre a checagem do cache e o request), uma única retry resolveria sem expor erro ao usuário.
- **Correção sugerida:** Adicionar retry automático (1x) no 401:
```typescript
if (response.status === 401) {
  this.cachedToken = null;
  // Retry once with fresh token
  const freshHeaders = await this.getAuthHeaders();
  const retryResponse = await fetch(url, {
    ...options,
    headers: { ...freshHeaders, ...(options.headers as Record<string, string>) },
    dispatcher: this.getHttpsAgent(),
  });
  if (retryResponse.status === 401) {
    throw new Error(`Santander API: Não autorizado (401)...`);
  }
  return retryResponse;
}
```

#### Finding 4
- **Severity:** 🔵 INFO
- **Confidence:** 0.90
- **Arquivo:** `santander.provider.ts` — `constructor` (~linha 215)
- **Descrição:** O construtor aceita `_webhookSecret` mas nunca usa. O Santander não suporta HMAC, a validação é por URL. O parâmetro está lá por compatibilidade com a factory, mas o `_` prefix já indica unused — está OK, mas vale um JSDoc explicando que Santander não usa webhook secret.
- **Correção sugerida:** Adicionar comentário no construtor:
```typescript
/** @param _webhookSecret - Unused. Santander validates webhooks by unique URL per provider, not HMAC. */
```

#### Finding 5
- **Severity:** 🟡 WARN
- **Confidence:** 0.70
- **Arquivo:** `santander.provider.ts` — `validateWebhook` (~linha 305)
- **Descrição:** A validação de webhook é muito superficial: só checa Content-Type e JSON válido. Qualquer pessoa que conheça a URL do webhook pode enviar payloads falsos. A proteção é apenas security by obscurity (providerId na URL). Se o providerId vazar (ex: logs, URL compartilhada), um atacante pode simular pagamentos.
- **Correção sugerida:** Considerar adicionar validação extra:
  1. IP allowlist do Santander (se documentado)
  2. Verificação de campos obrigatórios no payload (covenantCode deve bater com o provider configurado)
  3. Após processar webhook, fazer GET de confirmação na API do Santander para validar o status real antes de marcar como PAID

#### Finding 6
- **Severity:** 🔵 INFO
- **Confidence:** 0.85
- **Arquivo:** `santander-sequence.ts` — `getNextBankNumber` (~linha 25)
- **Descrição:** Usa `Prisma.TransactionIsolationLevel.Serializable` que é correto para atomicidade, mas pode gerar deadlocks sob carga alta. Com Serializable, duas transações concorrentes no mesmo `companyId+covenantCode` vão serializar (uma retry), o que é o comportamento esperado. Porém, o código não tem retry em caso de serialization failure (`P2034`).
- **Correção sugerida:** Adicionar retry para `PrismaClientKnownRequestError` com code `P2034`:
```typescript
const MAX_RETRIES = 3;
for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
  try {
    return await prisma.$transaction(async (tx) => { ... }, { isolationLevel: ... });
  } catch (err) {
    if (err instanceof Prisma.PrismaClientKnownRequestError && err.code === 'P2034' && attempt < MAX_RETRIES - 1) {
      continue;
    }
    throw err;
  }
}
```

#### Finding 7
- **Severity:** 🔵 INFO
- **Confidence:** 0.90
- **Arquivo:** `santander.provider.ts` — `createBoleto` (~linha 245)
- **Descrição:** O `generateNsuCode()` usa `Date.now().toString(36) + Math.random().toString(36)`. Em ambiente de alta concorrência (ex: geração em lote), dois boletos criados no mesmo milissegundo podem ter timestamp idêntico. A parte random (`5 chars base36`) dá ~60M combinações, risco baixo mas não zero.
- **Correção sugerida:** Considerar usar `crypto.randomUUID().replace(/-/g, '').slice(0, 20)` para unicidade garantida. Baixa prioridade.

#### Finding 8
- **Severity:** 🔵 INFO
- **Confidence:** 0.85
- **Arquivo:** `webhooks/santander/[providerId]/route.ts` — (~linha 75)
- **Descrição:** O webhook route sempre passa `sandbox: false` na instanciação do gateway: `{ sandbox: false, companyId: provider.companyId }`. Se o provider estiver configurado como sandbox, o webhook vai criar o gateway apontando para produção, o que pode causar inconsistência nas URLs usadas internamente (embora no contexto de webhook isso não faça requests de saída, então o impacto real é mínimo).
- **Correção sugerida:** Usar o valor real do provider:
```typescript
{ sandbox: provider.sandbox, companyId: provider.companyId }
```

#### Finding 9
- **Severity:** 🟡 WARN
- **Confidence:** 0.80
- **Arquivo:** `boletos/[boletoId]/pdf/route.ts` — (~linha 65-70)
- **Descrição:** O endpoint de PDF faz `parseInt(payerDocumentNumber, 10)` dentro do `getBankSlipPdf`. Para CNPJs longos (14 dígitos), `parseInt` em JavaScript pode perder precisão para números > `Number.MAX_SAFE_INTEGER` (2^53). Um CNPJ como `12345678901234` = 12.3 trilhões, que está dentro do safe integer range. Porém, é uma prática frágil — o Santander espera número ou string? Se espera string, o `parseInt` é desnecessário e potencialmente destrutivo.
- **Descrição adicional:** Olhando `santander.provider.ts` `getBankSlipPdf` linha ~380: `payerDocumentNumber: parseInt(payerDocumentNumber, 10)`. Converte string para number no body JSON. Se o Santander aceita string, manter como string é mais seguro.
- **Correção sugerida:** Verificar a documentação da API Santander. Se aceitar string:
```typescript
body: JSON.stringify({
  payerDocumentNumber: payerDocumentNumber, // manter como string
}),
```

#### Finding 10
- **Severity:** 🔵 INFO
- **Confidence:** 0.75
- **Arquivo:** `page.tsx` (frontend) — componente `PemCertificateField`
- **Descrição:** O upload de certificado lê o arquivo inteiro como texto e armazena no state. Para certificados normais (~2-4KB) está OK. Porém não há validação de tamanho máximo do arquivo — um usuário poderia fazer upload de um arquivo grande por engano.
- **Correção sugerida:** Adicionar check de tamanho antes de ler:
```typescript
if (file.size > 50_000) { // 50KB mais que suficiente para cert PEM
  toast.error("Arquivo muito grande. Certificados PEM normalmente têm menos de 10KB.");
  return;
}
```

#### Finding 12
- **Severity:** 🟡 WARN
- **Confidence:** 0.70
- **Arquivo:** `santander.provider.ts` — `createBoleto` (~linha 260)
- **Descrição:** Se `input.customer.address` é `undefined`, vários campos do payer recebem valores placeholder ("Não informado", "SP", "00000-000"). O Santander pode rejeitar boletos com CEP `00000-000` ou endereço inválido, pois o BACEN exige dados reais do pagador para registro de boleto. Isso pode funcionar em sandbox mas falhar em produção.
- **Correção sugerida:** Considerar validar endereço completo antes de enviar, ou pelo menos logar warning quando usando placeholders:
```typescript
if (!input.customer.address?.zipCode) {
  console.warn(`[santander] Boleto criado sem CEP do pagador — pode ser rejeitado pelo Santander em produção`);
}
```

### Checklist

- [x] **Segurança** — Certificados tratados como `type: "password"` no registry, nunca logados. Webhook validation é básica (Content-Type + JSON) mas tem security by obscurity via providerId na URL. Sem secrets hardcoded. Credentials encriptadas em DB. `rejectUnauthorized: true` no mTLS.
- [x] **Funcionalidade** — Cobre todos os flows: create, status, cancel, webhook, PDF. Edge cases de endereço incompleto cobertos com placeholders (ver Finding 12). Boleto híbrido (barras + Pix) implementado corretamente com txId.
- [x] **Code Smells** — Código bem organizado. Helpers isolados, types definidos. Alguma duplicação nos patterns de AbortController/timeout (aceitável). Funções utilitárias poderiam ser extraídas para um módulo compartilhado no futuro.
- [x] **Error Handling** — Erros bem tratados com mensagens claras e contexto. Settlement fetch é best-effort (não quebra status query). Timeout com AbortController. Webhook route retorna 200 para evitar retries infinitos do Santander.
- [x] **TypeScript** — Tipos bem definidos. Uso correto de `as const`, interfaces, narrowing. `@ts-expect-error` justificado para `dispatcher` do undici.
- [x] **Prisma** — Migration correta com unique index, FK com CASCADE, `@updatedAt`. Sequência atômica com Serializable isolation. Schema alinhado com migration.
