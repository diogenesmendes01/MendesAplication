## Findings

### 🟡 WARN-01: JSDoc desatualizado no `SantanderProvider.getBankSlipPdf`

**Arquivo:** `src/lib/payment/providers/santander.provider.ts` (linha ~469)
**Problema:** O comentário do método diz:
> "This is a Santander-specific method - NOT part of PaymentGateway interface."

Mas agora **é** parte da interface (`PaymentGateway.getBankSlipPdf?`). A PR #314 adicionou o método à interface em `types.ts` mas não atualizou o JSDoc do provider que implementa.

**Impacto:** Apenas documentação — zero impacto funcional. Pode confundir dev futuro.
**Ação:** Atualizar JSDoc pra remover "NOT part of PaymentGateway interface" e substituir por algo como "Implements PaymentGateway.getBankSlipPdf".

### ✅ INFO-01: Factory — equivalência de comportamento confirmada

**Arquivo:** `src/lib/payment/factory.ts`

O switch/case original foi substituído por `GATEWAY_FACTORIES` (registry map). Validei que:

- **mock** → `new MockProvider()` ✅
- **pagarme** → valida `apiKey`, instancia com credentials + metadata + webhookSecret ✅
- **pinbank** → throw "não implementado" (mesmo comportamento anterior) ✅
- **santander** → monta `SantanderCredentials`, passa todos os 6 args ✅

Dupla validação (PROVIDER_REGISTRY + GATEWAY_FACTORIES) é intencional — separa "provider existe no sistema" de "provider tem implementação de gateway". Correto.

Em produção, `mock` não está em `PROVIDER_REGISTRY` → `getGateway("mock")` falha com "Provider not found". Em dev, funciona. Comportamento esperado.

### ✅ INFO-02: Webhook handler — lógica equivalente verificada

**Arquivo:** `src/lib/payment/webhook-handler.ts`

O helper `processBoletoWebhookEvent` encapsula toda a lógica que estava inline na route do Santander. Confirmei:

- **Status mapping** (`WEBHOOK_TO_BOLETO_STATUS`) — mapeia os 4 event types corretamente:
  - `boleto.paid` → `PAID` ✅
  - `boleto.cancelled` → `CANCELLED` ✅
  - `boleto.expired` → `OVERDUE` ✅
  - `boleto.failed` → `CANCELLED` ✅
- **Busca por gatewayId** — suporta full match e partial (`%.suffix`) ✅
- **SELECT FOR UPDATE** — race condition prevention mantido ✅
- **Idempotência** — skip se já está no status alvo ✅
- **AccountReceivable** — FK direto + fallback heurístico (valor ± tolerância, dueDate ± window) ✅
- **Audit log** — fire-and-forget com `.catch()` ✅

### ✅ INFO-03: Route do webhook — refatoração limpa

**Arquivo:** `src/app/api/webhooks/santander/[providerId]/route.ts`

A route agora é um orquestrador fino:
1. Lê body → 2. Valida provider → 3. Instancia gateway → 4. Valida webhook → 5. Parseia evento → 6. Delega pro handler → 7. Responde 200

Todos os status codes mantidos (200 pra ACK, 400 pra webhook inválido, 500 pra erro de body). Sem regressão.
