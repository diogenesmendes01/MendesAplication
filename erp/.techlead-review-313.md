### Avaliação de Arquitetura

#### ✅ Aderência aos padrões do projeto

A integração segue fielmente os padrões estabelecidos:

- **PaymentGateway interface:** SantanderProvider implementa todos os 6 métodos do contrato (`createBoleto`, `getBoletoStatus`, `cancelBoleto`, `validateWebhook`, `parseWebhookEvent`, `testConnection`). Assinaturas e tipos de retorno 100% conformes.
- **Registry pattern:** Santander registrado em `PRODUCTION_PROVIDER_REGISTRY` com `configSchema` e `settingsSchema` seguindo a mesma estrutura de Pagar.me e PinBank.
- **Factory pattern:** Switch case em `factory.ts` segue o mesmo padrão dos outros providers, com validação de credenciais antes de instanciar.
- **Constants:** `PRODUCTION_PROVIDER_TYPES` atualizado, o type system (`ProviderType`) se atualiza automaticamente via inferência. Nenhuma mudança necessária em `types.ts`.

#### ✅ Extensibilidade e manutenibilidade

**Excelente separação de responsabilidades** — melhor que o PagarmeProvider (monolítico):

| Módulo | Responsabilidade | LOC aprox |
|--------|-----------------|-----------|
| `santander-auth.ts` | mTLS + OAuth token lifecycle | ~180 |
| `santander-sequence.ts` | Geração atômica de nosso número | ~80 |
| `santander.provider.ts` | Lógica de negócio (interface) | ~550 |

Cada módulo é testável isoladamente. O `SantanderAuthManager` encapsula toda a complexidade de mTLS/OAuth, o que facilita mock em testes e reuso se o Santander tiver outras APIs no futuro.

**Sequence generator:** Solução robusta com `Serializable` isolation level + retry no `P2034`. Padrão correto para operações de incremento atômico com Prisma/PostgreSQL.

**Schema:** `SantanderSequence` é um modelo dedicado com composite unique constraint `[companyId, covenantCode]`. Limpo, não polui o schema existente. `@@map("santander_sequences")` segue convenção do projeto.

#### ✅ Decisões técnicas sólidas

1. **mTLS via undici.Agent:** Decisão correta. O `fetch` nativo do Node.js 18+ é implementado sobre undici e ignora `https.Agent`. Usar `dispatcher` é a forma correta. Os `@ts-expect-error` são aceitáveis e bem documentados.

2. **Token caching com margem de 60s:** Padrão OAuth bem implementado. Renovação proativa evita requests com token prestes a expirar.

3. **Retry automático no 401:** `authenticatedFetch` invalida cache e retenta 1x. Sem risco de loop. Trata 403 no retry também.

4. **gatewayId composto:** `nsuCode.nsuDate.ENV.covenantCode.bankNumber` — encoding pragmático que permite tanto lookup exato quanto busca por sufixo nos webhooks. Formato determinístico, parseable com `split(".")`.

5. **Webhook por URL única:** Santander não oferece HMAC. Autenticação via `providerId` na URL + validação de `covenantCode` no payload é a abordagem correta para esse cenário. O parâmetro `_webhookSecret` está documentado como unused.

6. **Webhook defensivo:** `parseWebhookEvent` tenta múltiplos nomes de campo (`status/situacao/eventType`, `bankNumber/nossoNumero`, etc.) porque a documentação do Santander é incompleta. Abordagem pragmática e correta.

### Findings

#### 🟡 ARCH-WARN-01 — Método `getBankSlipPdf` fora da interface `PaymentGateway`

**Onde:** `santander.provider.ts` (método extra) + `boletos/[boletoId]/pdf/route.ts` (instanceof check)

O método `getBankSlipPdf` é Santander-specific e não faz parte do `PaymentGateway`. O route usa `instanceof SantanderProvider` para acessá-lo:

```typescript
if (!(gateway instanceof SantanderProvider)) {
  return NextResponse.json({ error: "Provider não suporta download de PDF" }, ...);
}
```

**Por que é WARN e não BLOCK:** Hoje só Santander precisa de PDF download via API. Se mais providers precisarem, será natural estender a interface. Forçar agora seria over-engineering.

**Recomendação futura:** Se um segundo provider precisar de PDF download, extrair para a interface:
```typescript
// Opcional na interface
getBankSlipPdf?(covenantCode: string, bankNumber: string, payerDoc: string): Promise<{ link: string }>;
```

#### 🟡 ARCH-WARN-02 — Lógica de atualização boleto/receivable duplicada no webhook route

**Onde:** `webhooks/santander/[providerId]/route.ts`

A lógica de transação que atualiza boleto status + AccountReceivable (incluindo fallback heurístico para receivables legados) parece ser duplicada do webhook route genérico (provavelmente existe um para Pagar.me). O Santander route reimplementa:
- SELECT FOR UPDATE + idempotência
- Lookup por boletoId FK → fallback heurístico
- Atualização de AccountReceivable
- Audit log

**Recomendação:** Extrair para helper compartilhado (`lib/payment/webhook-handler.ts`):
```typescript
export async function processBoletoWebhookEvent(
  event: WebhookEvent,
  provider: PaymentProvider,
): Promise<WebhookProcessResult> { ... }
```

Isso elimina duplicação e garante consistência entre providers. Não é blocker porque a lógica está correta e testada, mas é tech debt que vai acumular com cada novo provider.
