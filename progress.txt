## US-006 — Webhook route + baixa automática no financeiro (2026-03-10) ✅

### What was implemented
- `src/app/api/webhooks/payment/[provider]/route.ts` — Next.js App Router POST handler
- Identifies provider type from path param, finds all active PaymentProviders of that type
- Decrypts credentials from each provider using `decrypt()` from encryption.ts, instantiates gateway via factory
- Tries to validate webhook signature with each provider until one matches (supports multiple providers of same type)
- Returns 401 if no provider validates the signature
- Parses webhook event via `gateway.parseWebhookEvent(rawBody)`
- Finds boleto by `gatewayId` + `providerId`
- Updates boleto status: boleto.paid→PAID, boleto.cancelled→CANCELLED, boleto.expired→OVERDUE, boleto.failed→CANCELLED
- If boleto paid: finds matching AccountReceivable by companyId + clientId (from proposal) + value (±R$0.01 tolerance) + dueDate (±15 days) + status PENDING, updates to PAID with paidAt
- Creates AuditLog with userId "system" for all operations (including boleto not found)
- Returns 200 OK to the bank even if boleto not found (avoids infinite retry loops)

### Learnings
- Next.js 15 App Router dynamic route params are now a Promise — must `await params` before destructuring
- AccountReceivable doesn't have a direct FK to Boleto — matching is done by companyId + clientId (from boleto.proposal.clientId) + value with tolerance + dueDate window
- Prisma Decimal fields accept plain numbers in `gte`/`lte` comparisons — no need for `new Prisma.Decimal()`
- Webhook handlers should be defensive: always return 200 for parse errors and missing boletos to prevent infinite retries from payment providers
- The `logAuditEvent` function is fire-and-forget (catches errors internally) — safe to use without try/catch
- For webhook routes with no user session, using "system" as userId in audit logs is the established pattern
