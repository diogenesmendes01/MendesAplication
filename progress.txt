## Contexto Inicial

### O que já existe no código
- `src/lib/ai/provider.ts` — suporta OpenAI, DeepSeek, Anthropic. Lê config de env vars globais (AI_PROVIDER, AI_API_KEY, AI_MODEL)
- `src/lib/ai/agent.ts` — agent loop com tools, funciona só para WhatsApp
- `src/lib/ai/tools.ts` — 6 tools: SEARCH_DOCUMENTS, GET_CLIENT_INFO, GET_HISTORY, RESPOND, ESCALATE, CREATE_NOTE
- `src/lib/ai/tool-executor.ts` — executa as tools, RESPOND envia via WhatsApp
- `src/lib/ai/embeddings.ts` — RAG com embeddings OpenAI (provider separado via AI_EMBEDDING_*)
- `src/lib/workers/ai-agent.ts` — BullMQ worker que checa AiConfig e roda o agent
- `src/lib/workers/whatsapp-inbound.ts` — enfileira no aiAgentQueue após criar TicketMessage
- `src/lib/workers/email-inbound.ts` — NÃO enfileira no aiAgentQueue (precisa adicionar)
- `src/app/(app)/configuracoes/ai/` — página de config com: toggle, persona, welcome message, keywords, max iterations
- `src/lib/encryption.ts` — encrypt/decrypt AES já existente (usado por canais de email e payment providers)

### Schema atual (AiConfig)
```prisma
model AiConfig {
  id                 String   @id @default(cuid())
  companyId          String   @unique
  enabled            Boolean  @default(false)
  persona            String   @db.Text
  welcomeMessage     String?  @db.Text
  escalationKeywords String[]
  maxIterations      Int      @default(5)
  createdAt          DateTime @default(now())
  updatedAt          DateTime @updatedAt
  company Company @relation(...)
}
```

### Padrões a seguir
- encrypt/decrypt: mesmo de `src/lib/encryption.ts` (AES-256-GCM)
- Server actions: requireCompanyAccess + requireAdmin + logAuditEvent
- Mascarar segredos: últimos 4 chars visíveis (padrão de payment-providers US-008)
- Migration: manual SQL se shadow DB não for acessível (padrão do projeto)
- DB via SSH tunnel: `ssh -f -N -L 15432:10.0.1.10:5432 root@coolify.boletoapi.com`

## US-002 — Refatorar provider.ts multi-provider (2026-03-15)

### O que foi feito
- chatCompletion() agora recebe ProviderConfig como 3º parâmetro obrigatório (não lê mais process.env)
- Retorno AiResponse inclui usage: { inputTokens, outputTokens } extraído da resposta da API (OpenAI-compat e Anthropic)
- 5 providers suportados: openai, anthropic, deepseek, grok, qwen
- Grok usa base URL https://api.x.ai (OpenAI-compatible)
- Qwen usa base URL https://dashscope.aliyuncs.com/compatible-mode (OpenAI-compatible)
- Default models adicionados: grok='grok-2', qwen='qwen-max'
- Helper getEnvProviderConfig() criado para backward compat (lê AI_PROVIDER/AI_API_KEY/AI_MODEL)
- agent.ts atualizado: usa apiKey do AiConfig se disponível, senão faz fallback para getEnvProviderConfig()
- npm run lint passa sem erros

### Lições
- ⏳ Anthropic retorna usage em data.usage.input_tokens/output_tokens; OpenAI-compat usa data.usage.prompt_tokens/completion_tokens
- ⏳ O ChatCompletionOptions interno foi eliminado — ProviderConfig cobre tudo (provider, apiKey, model, temperature, maxTokens)
- ⏳ embeddings.ts não foi tocado (usa AI_EMBEDDING_* separadamente, conforme spec)

## US-003 — Cost tracker + model suggester (2026-03-15)

### O que foi feito
- Criado src/lib/ai/cost-tracker.ts com: logUsage(), getTodaySpend(), getUsageSummary()
- Criado src/lib/ai/model-suggester.ts com: suggestModel()
- Tabela MODEL_PRICING cobre 10 modelos: gpt-4o, gpt-4o-mini, claude-opus-4-20250514, claude-sonnet-4-20250514, claude-haiku-4-20250414, grok-2, grok-2-mini, qwen-max, qwen-plus, qwen-turbo
- BRL_USD_RATE = 5.80 como constante nomeada exportada

- suggestModel retorna ModelSuggestion { model, estimatedDailyCostBrl } — o modelo mais potente que cabe no orçamento

- npm run lint passa sem erros

### Lições
- ⏳ Prisma Decimal precisa ser instanciado com `new Decimal(value.toFixed(N))` para controlar precision
- ⏳ groupBy do Prisma retorna _sum como Decimal | null — sempre usar Number() para converter
- ⏳ MODEL_PRICING e BRL_USD_RATE exportados do cost-tracker para reuso no model-suggester (evita duplicação)

## US-004 — Adaptar agent.ts para multi-canal + provider config (2026-03-15)

### O que foi feito
- runAgent() agora recebe parâmetro channel: 'WHATSAPP' | 'EMAIL' (default 'WHATSAPP' para backward compat)
- Descriptografa apiKey via decrypt() de encryption.ts antes de montar ProviderConfig
- Fallback para getEnvProviderConfig() quando empresa não tem apiKey configurada
- Verifica whatsappEnabled/emailEnabled antes de processar — retorna error específico se desabilitado
- Verifica dailySpendLimitBrl via getTodaySpend() — retorna 'daily_spend_limit_reached' sem chamar LLM
- Usa emailPersona quando channel=EMAIL e emailPersona não é null, senão usa persona padrão
- System prompts separados: buildWhatsAppSystemPrompt() (conciso, direto) e buildEmailSystemPrompt() (formal, inclui assinatura)
- Cada chamada a chatCompletion() tem logUsage() logo após, registrando tokens no AiUsageLog
- Terminal actions expandidas: RESPOND e RESPOND_EMAIL ambos param shouldStop=true
- ai-agent.ts worker passa channel do job data para runAgent()
- whatsapp-inbound.ts inclui channel: 'WHATSAPP' no job data do aiAgentQueue

### Lições
- ⏳ Decimal do Prisma precisa de Number() para comparação (dailySpendLimitBrl é Decimal no DB)
- ⏳ O contact model do Prisma (AdditionalContact) tem campo email — útil para futura resolução de destinatário de email
- ⏳ O default do channel param garante backward compat — callers existentes não precisam mudar
